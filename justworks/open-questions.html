Customer Central MVP · Prompt Requirement #4

Open Questions & Dependencies

What we don't know, who can tell us, and what changes if the answer differs from our assumption. These questions are designed to be answered — many by the people in this room.

This proposal is built on 45 documented assumptions. Some are confirmed through public research. Most are inferred from domain patterns and the case study prompt. The questions below represent the assumptions with the highest impact if wrong — the ones that could change MVP scope, architecture decisions, or timeline. They're organized by who is best positioned to answer each one.

C
Caroline Crossland Salesforce Solutions Architect
Scope
Is the Zendesk Organization ID mapped to the Salesforce Account ID today — and if so, how reliable is that mapping?
▼
Why This Matters

The unified interaction timeline (FS-2) merges Zendesk tickets with Salesforce Cases into a single chronological view. This is the hardest integration in the MVP. Without a clean join key, we can't reliably connect a customer's support tickets to their account record.

Our Assumption

We assume a shared key exists — likely Zendesk Organization ID mapped to Salesforce Account ID, or a shared external ID field. Confidence: Medium. Most Salesforce-heavy orgs implement this, but the mapping quality varies.

If the Mapping Exists & Is Clean

FS-2 proceeds as scoped. The aggregation layer joins on this key. Timeline is unaffected.

If the Mapping Is Broken or Missing

Scope change. FS-2 degrades to Zendesk-only interaction history in MVP. We'd need a Week 1 spike to build or repair the mapping — potentially a multi-week data cleanup effort. This is the single highest-risk integration question.

Architecture
What custom Salesforce objects exist for Justworks-specific entities (Plans, Members, Product Configurations)?
▼
Why This Matters

Our data requirements reference assumed custom objects like Plan__c and Member__c. If the actual schema uses different structures — or if plan data lives in the production DB rather than Salesforce — it changes which systems we query and how we join data.

Our Assumption

Plan type and product configuration are stored as custom Salesforce objects or fields on the Account record. Confidence: Medium-Low. Justworks' dedicated Salesforce Delivery Team suggests significant customization, but the specific schema is unknown.

If Plan Data Lives Elsewhere

If plan configuration is primarily in the production DB (not Salesforce), the company profile display (FS-1, FS-3) pulls from a different source than expected. This simplifies some integrations (less Salesforce dependency for real-time data) but changes the aggregation layer design. Week 1 data access spike would reveal this.

Architecture
What is the current Salesforce API rate limit, and does an existing middleware or data warehouse sync layer exist?
▼
Why This Matters

Customer Central will query Salesforce on every call lookup. With thousands of daily interactions, direct real-time SOQL queries will hit rate limits. We need to know if there's already a middleware layer (MuleSoft, custom API, data warehouse sync) we should build on top of.

Our Assumption

We assume an aggregation layer that caches Salesforce data — not direct real-time queries. But if an existing sync pipeline or middleware exists, we should use it rather than building a parallel one.

Discovery Action

Week 1 spike: map existing data infrastructure. If a middleware layer exists, our aggregation service sits on top of it. If not, we build the sync pipeline as part of the data engineer's Week 1-2 work. Either way, we're not querying Salesforce live on every call.

Informational
Are all Justworks product lines (Payroll, PEO Basic, PEO Plus, EOR, International Contractors) in a single Salesforce org?
▼
Why This Matters

If products are in separate orgs, querying a customer's complete product relationship requires cross-org API calls — significantly more complex. If they're in one org with custom objects per product line, it's a simpler query with filters.

Our Assumption

Single org. Confidence: Medium-High. Most mid-size SaaS companies consolidate into one org. Justworks' dedicated Salesforce team suggests a mature, centralized setup.

E
Evan Lehrman Senior Technical Product Manager
Scope
Can the payroll engine data (pay stubs, run history, withholdings) be accessed via internal API, or does it require direct database queries?
▼
Why This Matters

PS-1 (Pay Stub Context) is the highest-volume problem story in the MVP — "Why is my paycheck different?" is the #1 call driver. The feature's value depends on accessing the last 3 pay stubs with line-item detail and delta highlighting. If the payroll engine doesn't expose this data cleanly, the feature either gets simplified or the integration timeline extends.

Our Assumption

Internal API exists for payroll data. Confidence: Medium. The Justworks product dashboard already displays pay stubs to customers, so an API likely exists — but it may not expose the granularity we need (line-item detail, period-over-period comparison).

If Clean API Exists

PS-1 builds as scoped: last 3 stubs, line-item breakdown, delta highlighting. This is the feature that makes agents say "I can't go back." On-time delivery.

If Only Raw Database Access

Scope change. PS-1 could degrade to a deep link to the existing payroll view (like PS-2) rather than an embedded experience. Or the data engineer adds a payroll data API to the Week 1-2 sprint, extending the build. Either way, we need to know in Week 1.

Timeline
What is the current data refresh latency from the production database to the data warehouse?
▼
Why This Matters

If the warehouse sync is batch (daily ETL), data in Customer Central could be up to 24 hours stale. An agent looking at "last payroll run" would see yesterday's data, not today's. For a live-call tool, that's unacceptable for some data points (billing status, open tickets) and tolerable for others (company profile, employee count).

Our Assumption

Mixed latency. The aggregation layer queries real-time sources (production DB via read replica, Zendesk API) for time-sensitive data and warehouse for stable data. We show freshness timestamps on every data point so agents know what they're looking at.

Architectural Response

The loading tier design already accounts for this: Tier 1 data (identity, status) must be near-real-time. Tier 3-4 data (benefits detail, pay stubs) can tolerate minutes of lag. Freshness indicators on the UI tell agents exactly when data was last updated — stale data is worse than no data only if the agent doesn't know it's stale.

Architecture
What is the billing system architecture — internal service, Stripe, or another payment processor?
▼
Why This Matters

FS-3 (Account Status Strip) and PS-6 (Payment Failure Cascade) both depend on billing data — payment status, outstanding balance, failed payment history. The integration approach differs dramatically between an internal billing service (direct API) and a third-party processor (webhook + API).

Our Assumption

Custom internal billing system, possibly with Stripe as the payment processor underneath. Confidence: Low. PEOs typically build custom billing due to the complexity of per-employee-per-month pricing across multiple product lines.

R
Renee Orser VP, Internal Tools & Operations
Scope
Does CSO handle all product lines (Payroll, PEO, EOR) with the same agents, or are there specialized teams per product?
▼
Why This Matters

If agents are segmented by product line, the dashboard can be more focused — a Payroll agent doesn't need PEO benefits data. If agents handle all products, the dashboard must surface context across all product lines for any caller, which increases data complexity and UI surface area.

Our Assumption

Agents handle multiple product lines. Confidence: Medium. The prompt says "across our product offerings" and the 24/7 coverage model suggests generalist agents with specialist escalation paths.

If Generalist Agents

Dashboard must show the company's full product relationship and adapt context based on which products they use. The FS-3 status strip is designed for this — it surfaces the most relevant signals regardless of product line.

If Specialized Teams

Opportunity to build product-specific dashboard views. Reduces UI complexity per agent but increases the number of views to design. The API-first architecture supports this — the data layer is the same, only the presentation changes.

Scope
What is CSO's current tooling workflow? Which systems do agents actually use and in what order?
▼
Why This Matters

Our entire problem framing — the 6 compounding problems, the context-switching tax, the 5-6 system toggle — is hypothesized based on typical support org patterns. If agents have already built effective workarounds, or if the pain is concentrated in different areas than we assume, the MVP priorities shift.

Our Assumption

Agents toggle between Zendesk, Salesforce, Justworks admin dashboard, internal knowledge base, Slack, and possibly Tableau. Context-switching is a primary pain point. Confidence: Medium-High. This pattern is near-universal in multi-product support orgs.

Day 1 Validation

First action in the role: shadow 5-10 CSO agents for 2-3 days. Watch what they actually do — which systems they open, in what order, where they hesitate, where they put customers on hold. Observed behavior beats self-reported pain. This validates or invalidates the core problem framing within the first week.

Timeline
What is the ITO team's current capacity? Are there competing initiatives that would affect resource allocation?
▼
Why This Matters

The engineering resource plan proposes 3 software engineers, 1 data engineer (from Data Services), and 1 designer. If the ITO team is already committed to other projects — Sales tooling, RevOps integrations — the timeline extends or the team composition changes.

Our Assumption

The role description says Customer Central is a priority initiative. We assume Renee can allocate a dedicated team for 5 weeks. But the JD also mentions Sales org tooling as part of ITO's mandate — those priorities may compete.

Informational
Are there PII or compliance constraints on what customer data can be surfaced in a consolidated internal view?
▼
Why This Matters

Customer Central consolidates data from multiple systems — including payroll, benefits, and personal information. If there are access control requirements (e.g., not all agents can see salary data, or benefits information requires HIPAA-compliant handling), the data layer needs permission filtering and audit logging from day one.

Our Assumption

CSO agents already have access to this data across individual systems. Customer Central doesn't expand the access surface — it consolidates it. But we've included audit logging in the data performance design as a precaution.

A
Annelise Bryda Staff Experience Designer
Architecture
What does a CSO agent's physical workspace look like — single monitor, dual monitor, screen size, browser vs. desktop app?
▼
Why This Matters

The information architecture assumes a single-screen, browser-based experience with progressive disclosure. If agents have dual monitors, the layout could be fundamentally different — status on one screen, detail on another. If they're on laptops (remote agents), screen real estate is constrained and the cognitive hierarchy becomes even more critical.

Our Assumption

Browser-based, single screen. The prototype is designed for ~1280px+ with responsive scaling. All UX decisions — tabs instead of side panels, progressive disclosure, the cognitive sequence flow — assume limited viewport. Confidence: Medium.

Scope
Are there existing design system components or patterns from other Justworks internal tools we should build on?
▼
Why This Matters

Building on an existing design system accelerates delivery and ensures consistency with other internal tools. Building from scratch gives design freedom but adds time and creates a parallel visual language. If the ITO team has established patterns, our prototype should adopt them.

Our Assumption

Some component library exists (React-based, given the confirmed tech stack) but may not have all the patterns we need. The prototype is designed as a starting point for discussion, not a final UI. Annelise would lead the design refinement in sprint 1.

If the actual data shows a different distribution — for example, if benefits questions or compliance issues outweigh payroll — the MVP feature prioritization changes. Pull Zendesk ticket tag distribution for the last 90 days. Group by topic category. This data should be available within the first 48 hours of the role and would either confirm or redirect the feature prioritization.

Informational
What telephony system does CSO use, and is there any existing CTI or screen-pop capability?
▼
Why This Matters

CTI screen-pop is deferred from MVP (the ⌘K search covers 95% of the value). But knowing the telephony vendor now informs whether Phase 2 CTI integration is a 2-week project or a 2-month project. If a partial screen-pop already exists, we might incorporate it into MVP at low cost.

Our Assumption

No existing CTI integration with customer data. Search-based lookup is the MVP approach. If CTI already exists and just needs to be wired to Customer Central, that changes the equation.

Informational
Who owns the Zendesk instance — CSO directly, or a shared ops team? Can we modify ticket fields and tags?
▼
Why This Matters

The Tiebreaker Principle calls for structured interaction records — tagged topics, resolution types, outcome codes — to feed Phase 3 AI. If we can add custom ticket fields in Zendesk, agents tag interactions as part of their natural workflow. If we can't modify Zendesk (shared ownership, change control), the structured data capture moves to Customer Central's own logging layer.

Our Assumption

CSO has influence over Zendesk configuration but may need to coordinate with a shared ops team. The aggregation layer captures structured data regardless — the question is whether Zendesk can also be enriched.

External Dependencies Blocking or Gating
Data Services Team Allocation
The data engineer is likely a shared resource across multiple teams. Their availability in Weeks 1-3 is critical — the Salesforce sync pipeline, Zendesk integration, and production DB read replica setup are all on their plate. If unavailable, the core team can build against mock data but the integration timeline extends.
Owner: Renee
Salesforce Schema Documentation
Caroline or her team needs to provide schema documentation or access to a sandbox for the Week 1 data access spikes. Without this, the engineering team is guessing at object names, field types, and relationship structures. This is a 2-hour investment that saves 2 weeks of trial and error.
Owner: Caroline
CSO Leadership Buy-In for Alpha Testing
Alpha testing requires 5 volunteer agents pulled from production queue for daily feedback sessions in Week 4. CSO leadership (Natalie Miranda's team) must approve this — it means slightly reduced coverage during alpha. The agents selected should span experience levels and product line familiarity.
Owner: Renee → Natalie Miranda
Design Resources
Annelise or a team member allocated for the full 5-week build. The prototype provides a starting point, but production design — component refinement, accessibility, edge case handling, responsive behavior — requires dedicated design time. Heaviest in Weeks 1-2 (layout decisions) and Week 4 (alpha feedback iteration).
Owner: Annelise
Production DB Read Replica Access
Customer Central must query production data (payroll runs, employee rosters, benefits elections) without impacting production database performance. This typically means a read replica. If one doesn't exist, the DB team needs to provision it — a 1-2 week setup that should start in Week 1.
Owner: Evan → DB Team

These questions are features, not bugs. Presenting them demonstrates that we know the difference between what we've validated and what we've assumed. Many can be answered by the people in this room — and several are designed as workshop inputs. The proposal is built to adapt: if the answers shift a priority or invalidate an assumption, the phased architecture absorbs the change without a restart.
