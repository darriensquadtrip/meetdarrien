Customer Central MVP · Prompt Requirement #2 & #6

Phased Roadmap & Rollout Plan

Three phases. Each earns the right to the next by proving accuracy, earning trust, and accumulating the structured data that makes the next capability possible.

Phase 1 · MVP
Unified Context
5 Weeks Build + 2 Weeks Rollout
Consolidated customer profile, interaction history, account status, and high-volume problem panels. Agents open one tool instead of six.
Phase 2 · Intelligence
Health & Action
8–12 Weeks Post-MVP
Composite health scoring, in-place actions, benefits life events, compliance guidance. The tool becomes the workspace, not just the reference.
Phase 3 · Proactive
Prediction & Prevention
Following Quarter
Proactive alerts, upsell intelligence, AI interaction summaries, agent copilot. CSO shifts from reactive support to proactive partnership.

Section 2 Week-by-Week Detail
Pre-Work
Week 0 — Discovery & Validation
Before writing a single line of code
▼
PM — First 5 Days

Shadow 5-10 CSO agents across shifts and product lines for 2-3 days. Watch what they actually do — which systems they open, where they hesitate, where they put customers on hold. Validate or invalidate the 6 compounding problems. Pull Zendesk ticket tag distribution for 90 days to confirm top call drivers. Meet with Caroline for Salesforce schema walkthrough. Meet with Natalie Miranda's team for CSO workflow mapping. Baseline current AHT, FCR, and agent satisfaction metrics.

Gate → Start Build

Must be true: Core problem framing validated (agents do context-switch across 4+ systems). Top 3 call drivers confirmed. Salesforce schema documented enough to start data access spikes. CSO leadership bought in to alpha testing plan.

Week 1
Data Access Spikes & Foundation
De-risk the hardest integrations before committing to scope
▼
Eng 1 — Search & Identity

Data access spikes for search vertical: (1) Production DB read replica — company profiles, member data, billing status. (2) Validate search query performance against 500K contacts. Build search endpoint + company profile API as runnable integration tests. Measure response times — Tier 1 must be <500ms. Prove the data layer works before touching UI.

Eng 2 — Panels & History

Data access spikes for panel vertical: (1) Zendesk API — ticket retrieval, join key to Salesforce. (2) Production DB — payroll/billing data pass-through. (3) Event stream maturity. Build timeline API as integration test, measure Zendesk latency. Scaffold app shell + component library in parallel.

Data Engineer

Design the aggregation schema. Normalized tables for companies, contacts, interactions, events, billing. Define the caching strategy (what's cached vs. queried live). Start the Salesforce sync pipeline — this is the long pole.

Designer

Refine the information architecture based on Week 0 agent observations. Finalize the cognitive sequence layout: WHO → CONTEXT → HISTORY → PROBLEM-SPECIFIC. Build component specs for the status strip and search interface.

Milestone → Data Access Validated

End of Week 1 decision: Which data sources are accessible? If the Zendesk↔Salesforce join key doesn't exist, FS-2 degrades to Zendesk-only history and we add a cleanup sprint later. If payroll engine API doesn't exist, PS-1 degrades to deep link. Scope adjusts based on reality, not assumptions.

Week 2
API Core & Search
The aggregation layer and the most critical user interaction
▼
Eng 1 — Search & Identity (End-to-End)

Build the full search vertical: ⌘K search endpoint + UI (type-ahead, result cards), company profile card (API + component), caller identity card, status indicator strip. Wire to live production DB data. End of week: an agent can search and see a real, complete profile in <500ms.

Eng 2 — Panels & History (End-to-End)

Build the full panels vertical: Interaction timeline (API + UI — merged Zendesk + Salesforce), recent activity feed, tabbed problem panels (pay stubs, tax docs, benefits, invoices, payment cascade). Each panel follows the same pattern: API call → skeleton → render. Wire to live Zendesk + prod DB data.

Data Engineer

Salesforce sync pipeline operational. Accounts, Contacts, Cases flowing into the aggregation layer. Begin Zendesk sync — tickets, users, satisfaction ratings. Data validation checks: are record counts matching? Are timestamps consistent?

Designer

Component library finalized — cards, badges, timeline items, tab navigation. Interaction spec for search: keyboard shortcuts, result ranking, disambiguation (admin vs. employee with same name). Edge case designs: no results, multiple matches, new company with no history.

Milestone → API Contract Locked

Frontend and backend agree on all API contracts. Swagger/OpenAPI spec published. Frontend can switch from mock data to live data seamlessly. Any contract change after this point requires a conversation.

Weeks 3–4
Feature Build Sprint
All 9 MVP features come online
▼
Eng 1 — Search & Identity

Week 3: Integrate Salesforce data into company profiles (from DE pipeline). Fuzzy search tuning, edge cases (companies with 500+ employees, new accounts with zero history). Week 4: Alpha feedback fixes — search ranking, profile accuracy, status strip caching under concurrent load. Performance optimization.

Eng 2 — Panels & History

Week 3: Polish timeline merge logic (Zendesk + Salesforce deduplication). Deep link handlers for Payroll Engine and Benefits portal. Auto-refresh via polling. Week 4: Alpha feedback fixes — data accuracy in panels, loading performance for heavy payloads (pay stubs at ~20KB), graceful degradation when sources are slow.

Data Engineer

Week 3: Zendesk sync fully operational. Production DB read replica queries for payroll/billing data. Event stream capture for recent activity. Week 4: Data validation — compare aggregated counts to source systems. Fix discrepancies. Freshness monitoring dashboard.

QA (Shared)

Begin systematic testing. Test each feature against acceptance criteria. Data accuracy validation — does what the dashboard shows match what Zendesk/Salesforce/production DB show? Performance testing — search <2s, dashboard load <3s. Edge cases: companies with 500+ employees, new accounts with zero history, data sync delays.

Milestone → Core Features Working

All 9 MVP features functional with real data from at least one test company. May have rough edges, but the core flow works: search → profile → history → problem panel. Ready for alpha.

Week 5
Alpha — 5 Agents, Daily Feedback
Real users, real calls, real friction
▼
PM

5 volunteer agents use Customer Central alongside their existing tools for one full week. Not replacing — supplementing. Daily 15-minute feedback standup: What worked? What was wrong? What was missing? PM sits with at least 2 agents during live calls to observe. Capture every "I had to switch to Zendesk because..." moment — those are Phase 1 gaps or bugs.

Both Engineers · QA

Hot-fix mode. Each engineer fixes issues in their vertical — Eng 1 handles search/profile bugs, Eng 2 handles timeline/panel bugs. Data accuracy issues get fixed same-day. Performance issues get priority. The goal: by end of week, alpha agents are choosing Customer Central over their current workflow for at least 50% of interactions.

Designer

Observe agent sessions (via Hotjar or screen share). Identify where the information hierarchy fails — what do agents look for that isn't where they expect it? Where do they scroll when they should click? Produce a prioritized list of UX refinements for the iterate sprint.

Alpha Gate → Proceed to Beta?

Must be true to proceed: (1) Data accuracy ≥ 95% — agents trust what they see. (2) Core flow works without crashing or hanging. (3) At least 3 of 5 agents say they'd use it over current tools. (4) No critical data gaps identified that would erode trust at scale. If gate fails: Add 1 week to fix critical issues before beta.

Week 6
Beta — Full Team (20-30 Agents)
Scale test + structured feedback
▼
PM + CSO Team Lead

One full pod (20-30 agents) uses Customer Central as their primary tool for all interactions. Structured feedback via survey (mid-week + end-of-week). Hotjar session recordings for UX pattern analysis. Measure: AHT comparison (beta agents vs. non-beta agents), search-to-context time, feature usage heatmap. Identify 2-3 Champions — power users who naturally adopt and can train others.

Both Engineers

Performance at scale. 20-30 concurrent users — does each vertical hold? Eng 1 validates search + profile caching under load. Eng 2 validates timeline + panel loading under load. Data freshness under concurrent sessions — auto-refresh every 5 minutes without conflicts.

Beta Gate → Proceed to Full Rollout?

Must be true: (1) Agent satisfaction ≥ 3.5/5.0 on beta survey. (2) ≥ 70% of beta agents using Customer Central as primary tool by end of week. (3) No data accuracy regressions. (4) Performance holds under concurrent load. (5) Champions identified and willing to support rollout. If gate fails: Extend beta 1 week with targeted fixes.

Week 7
Full Rollout + Training
All CSO agents, supported launch
▼
Training Delivery

3-layer training approach: (1) Live demo session (45 min, recorded) — walk through the core flow with real data. Show the "before and after" — old workflow vs. Customer Central. (2) Self-paced sandbox — test environment with sample data where agents can explore without risk. (3) Quick reference card — one-page PDF with keyboard shortcuts, panel descriptions, and "where to find X" guide.

Champions Program

2-3 beta Champions serve as peer trainers and first-line support. They sit with new users during their first live calls with Customer Central. They collect feedback directly and escalate to PM. They get a dedicated Slack channel for real-time questions. Champions are recognized by CSO leadership — this isn't volunteer labor, it's a growth opportunity.

Ongoing Support

Week 1-4 post-launch: Weekly office hours (30 min, drop-in) for questions and feedback. PM monitors adoption dashboard daily — who's using it, who isn't, and why. Follow up personally with low-adoption agents. Month 2+: Shift to monthly check-ins. Feature request intake opens for Phase 2 prioritization. Begin measuring AHT and FCR impact with enough data for statistical significance.

Section 3 Phase 2 & 3 — What Comes Next
Phase 2 Health & Action 8–12 Weeks Post-MVP
Composite Health Score
Weighted RAG status per account. Configurable signals. Requires 60+ days of validated Phase 1 data.
In-Place Ticket Actions
Create and update Zendesk tickets without leaving Customer Central. Requires proven read-path trust.
Benefits Life Event Flows
Guided workflows for marriage, baby, divorce. Requires carrier API mapping and Benefits team partnership.
Proactive Alert System
Rule-based flags for at-risk accounts. Requires 1+ quarter of structured interaction data for baselines.
CTI Screen-Pop
Auto-populate customer profile when phone rings. Requires telephony vendor API integration.
Leadership Reporting
Aggregate health dashboards for team leads and VP level. Data already flowing to warehouse via API-first architecture.
Phase 3 Prediction & Prevention Following Quarter
Upsell & Cross-Sell Intelligence
Contextual product suggestions. Requires CSO/Sales/RevOps alignment and historical conversion data.
AI Interaction Summaries
NLP-generated summaries of past conversations. Trained on Phase 1's structured interaction records.
Interaction Intelligence
Trending issues, sentiment analysis, resolution path optimization across the full interaction corpus.
Agent Copilot
Real-time AI assistance during live calls. The culmination — Customer Central becomes the copilot, not just the workspace.
