Customer Central MVP · Prompt Requirement #6

Rollout & Training Plan

From 5 alpha agents to full CSO adoption in 3 weeks. A pull-based rollout that earns adoption through a clearly better tool — not a mandate.

3 Stages 5 Gate Criteria 3 Training Layers 2–3 Champions
Section 1 Rollout Philosophy

Before we build, it must be clear that what we are building is much better than the current solution. If we've done our job right, we won't need to push adoption — we'll have eager testers who see the difference immediately. They adopt, they help us make tweaks, and then they train the others. The rollout isn't about convincing people to use something new. It's about proving the tool is so much better that agents want to switch — then getting out of their way.

The rollout strategy is pull, not push. Build something clearly better → find your first believers → let them refine it → let them spread it. Forced adoption kills internal tools. Earned adoption compounds.

Section 2 The 3-Stage Rollout
Alpha · Week 5
First Believers
5 hand-picked agents
Agents who feel the pain most. If it's clearly better for them, they become our first believers. Alongside existing tools — supplementing, not replacing.
Beta · Week 6
Pod Expansion
1 full pod (20–30 agents)
Alpha believers help us refine. Structured surveys, AHT comparison, feature heatmap. Champion identification — the agents who naturally lean in hardest.
Full Rollout · Week 7
All CSO Agents
Full team supported launch
Champions (the early believers) train the rest. 3-layer training support. Old tools stay accessible for 30 days — no agent is ever without a working tool.
Section 3 Stage Gates
Pre-Alpha
Gate → Start Alpha
Must be true before any agent touches the tool
▼
  • Core problem framing validated — agents context-switch across 4+ systems (confirmed via shadowing)
  • Top 3 call drivers confirmed — Zendesk ticket tag distribution pulled for 90 days
  • Salesforce schema documented — enough to build data access spikes
  • CSO leadership bought in — Renee/Director level aware and supportive of alpha plan
  • Health score validated against agent judgment — before shipping the score, we validate it against experienced agent intuition. PM manually computes scores for 50-100 accounts using the weighted formula, experienced agents independently rate the same accounts, and we require ≥ 80% alignment before the score goes live
Gate Logic

All five criteria must be met. This gate exists to prevent building the wrong thing. If shadowing reveals a different problem than expected, the product scope changes before code is written. If health score validation falls below 80% alignment, weights are recalibrated before shipping.

Alpha → Beta
Gate → Expand to Pod
5 agents have used the tool for 1 full week
▼
  • Data accuracy ≥ 95% — agents trust what they see (validated against source systems)
  • Core flow works without crashing — search → profile → history → problem panel
  • ≥ 3 of 5 agents prefer it over current tools for majority of interactions
  • No critical data gaps — nothing missing that would erode trust at scale

If gate fails: +1 week to fix critical issues. PM diagnoses root cause (data accuracy? performance? UX?) and engineers hot-fix. Re-evaluate at end of extension week.

Beta → Full
Gate → Full Rollout
20-30 agents have used the tool as their primary tool for 1 week
▼
  • Agent satisfaction ≥ 3.5/5.0 on structured beta survey
  • ≥ 70% of beta agents using Customer Central as primary tool by end of week
  • No data accuracy regressions from alpha baseline
  • Performance holds under concurrent load — 20-30 simultaneous users, no degradation
  • Champions identified and willing — 2-3 power users ready to support rollout

If gate fails: Extend beta 1 week with targeted fixes. Champions continue collecting feedback. If satisfaction is below 3.0, escalate to leadership with honest assessment and revised timeline.

Section 4 Training Plan — 3 Layers
Layer 1
Live Demo Session
45 minutes, recorded — facilitator: PM + 1 Champion
▼
Structure

5 min — Context: why this exists, what problem it solves
20 min — Live walkthrough with real data
10 min — Q&A
10 min — Hands-on: agents try it themselves

Demo Flow

Cmd+K Search → Company Profile → Health Score Badge → History Timeline → Pay Stub Viewer → Billing Risk Cascade. The health badge shows a composite score from 5 signals + tenure adjustment — billing, payroll, support, satisfaction, and activity — weighted, combined, and adjusted by account tenure into a single Green/Amber/Red indicator. Tenure Factor: <30 days = 0.75, 30-90 days = 0.85, 90-180 days = 0.95, 180+ days = 1.00. Each step shows the "before and after" — old workflow vs. Customer Central.

Details

"Before and after" comparison is the centerpiece — show the agent doing a common task the old way (4 tabs, 3 systems, 90 seconds), then the new way (1 search, 15 seconds). Include health score: before, an agent checks billing, payroll, support tickets, and CSAT separately to gauge account health — 5 systems, 3 minutes. After, a single health badge in the status strip shows the composite score with a click-to-expand breakdown of all 5 signals + tenure adjustment. Recording available for async viewing. Sessions run in groups of 10-15 agents to keep Q&A productive.

Layer 2
Self-Paced Sandbox
Test environment with sample data — risk-free exploration
▼
Environment

Test environment pre-loaded with sample data (Acme Corp, same data as prototype). No impact on real customer data. Agents can explore, break things, and build muscle memory without risk.

Guided Tasks

"Find Sarah Chen's last interaction"
"Check Marcus Johnson's pay stub"
"View Acme's billing status"
Each task maps to a core user story, building familiarity with the real workflow.

Availability

Sandbox stays available for 2 weeks post-training. Agents can return any time to practice. After 2 weeks, agents have enough real usage that the sandbox is no longer needed.

Quick Reference Card
1-page PDF — printed and posted at each workstation
▼
Keyboard Shortcuts

Panel Guide

Common lookup scenarios mapped to UI location: "Client's last call?" → History tab, filter by type. "Is payroll running?" → Overview status strip. "Why was an invoice late?" → Billing tab, payment cascade. "Employee's benefits status?" → Benefits tab, coverage summary. "How healthy is this account?" → Health score badge in status strip, click for signal breakdown.

Section 5 Champions Program

Champions aren't assigned — they self-select. If the tool is clearly better, some agents will naturally lean in harder, give more feedback, and start showing teammates. Those are your champions. You don't recruit them. You recognize them.

Selection

2-3 power users identified during beta — highest adoption rate, most feedback submitted, most enthusiasm in standup sessions. They emerge organically. PM confirms willingness and formalizes the role.

Role

Peer trainers and first-line support. They train the rest because they believe in the tool. They sit with new users during first live calls. They collect feedback directly and escalate to PM.

Support

Dedicated Slack channel for real-time questions. Champions are the first responders — if they can't answer, PM steps in. Available during all shift hours for the first 4 weeks post-launch.

Recognition

Acknowledged by CSO leadership — this is a growth opportunity, not volunteer labor. Champions are highlighted in team meetings. Role transitions to feature advocate after 4 weeks post-launch.

Team Lead & Leadership Training

Leadership portfolio health view walkthrough — dedicated session for CSO team leads and leadership on the portfolio health list view. Shows how to read the aggregated health scores across their book of accounts, identify at-risk customers before they escalate, and use the signal breakdown to prioritize proactive outreach. Separate from agent training — this is a strategic tool, not a call-handling tool. Emphasize: health scores measure customer reality, not agent performance.

Section 6 Communication Plan
Timeline
Who Hears What, When
Building anticipation through transparency — no surprises
▼
  • Week 3 CSO leadership briefed on upcoming tool. Renee/Director level — set expectations for alpha timeline and what success looks like.
  • Week 4 Alpha agents personally invited by PM. 1-on-1 context setting — why they were chosen, what to expect, how their feedback shapes the tool.
  • Week 5 Alpha results shared with broader team. Build anticipation: "5 of your teammates tested a new tool — here's what they said." No hype, just honest results.
  • Week 6 Beta team announcement — "your pod is next." Team lead co-delivers the message. Agents know what's coming and why.
  • Week 7, Day 1 All-hands announcement + live demo. Champion co-presents with PM. Show real results from alpha and beta — real agent quotes, real time savings.
  • Week 7, Day 2–3 Small-group training sessions — 10-15 agents per session. Live demo + sandbox time. Champions available for questions.
  • Week 7, Day 5 Office hours — 30-minute drop-in for any remaining questions. PM + Champions available. Low-pressure, high-support.
Section 7 Adoption Metrics & Monitoring
Daily Dashboard

Active users, sessions per agent, features used, search queries. PM monitors daily during rollout weeks. This isn't surveillance — it's knowing where agents need help before they ask.

Targets
Target Adoption Curve
Expected adoption % at each milestone — realistic, not aspirational
▼
Milestone Target What It Means
Week 5 (Alpha) 100% 5/5 agents using alongside existing tools
Week 6 (Beta) 70%+ 20+/30 agents using as primary tool
Week 7 + 1 60% Majority of CSO agents onboarded and active
Week 7 + 2 80% Late adopters coming on after seeing peer success
Month 2 90%+ Sustained daily usage — Customer Central is the default
Intervention Trigger

If any agent hasn't logged in after 3 days post-training, PM follows up personally. Not punitive — diagnostic. "What's blocking you? What do you need?"

Anti-Metric

Don't optimize for login count alone. Track whether agents are actually faster (AHT) and resolving more (FCR). A tool that gets opens but doesn't improve outcomes is a distraction, not a solution.

Section 8 Risk Mitigation & Rollback
Risk 1
Agents don't trust the data
If the dashboard shows something different from Zendesk/Salesforce, agents stop using it
▼
Mitigation

Alpha tests with real production data — not test data. Data accuracy is an explicit gate criterion (≥ 95%). Daily validation during alpha: compare dashboard output to source systems for 10 random accounts.

Rollback

Fix data issues before proceeding. If accuracy is below 95% after alpha week, delay beta by 1 week. Data engineer prioritizes sync pipeline fixes. No agent is exposed to bad data at scale.

Risk 2
Tool is slower than current workflow
If Customer Central adds time to calls, agents will revert
▼
Mitigation

Performance testing in beta — AHT comparison between beta agents and non-beta agents. Search must return in <2s, dashboard load in <3s. Performance under 20-30 concurrent users validated before full rollout.

Rollback

Old tools stay accessible in parallel during rollout. If any agent finds the tool slower for a specific workflow, they use the old tool for that workflow while we fix it. No forced switching.

Risk 3
Resistance to change
"I know my current tools" — familiarity bias is real
▼
Mitigation

Champions as peer advocates. Hearing "this is better" from a teammate who does the same job is 10x more convincing than hearing it from PM. "Before and after" demo shows concrete time savings. No forced adoption initially — let the tool earn its place.

Rollback

Agents can use both tools during the 30-day transition period. Old tools remain fully accessible. If an agent prefers the old workflow for specific tasks, that's valid feedback — we improve the new tool, not mandate compliance.

Risk 4
Leadership loses patience
Adoption curves take time — leadership needs context, not just numbers
▼
Mitigation

Set expectations upfront. Share adoption curve targets with Renee before launch — "here's what 60% at Week 1 looks like, here's what 90% at Month 2 looks like." Weekly adoption reports to leadership with context, not just numbers. Include agent quotes, before/after comparisons, and concrete AHT improvements.

Full Rollback Procedure

If satisfaction drops below 3.0/5.0 at any point post-launch, pause rollout, diagnose, and fix. Old tools remain accessible throughout. No agent is ever without a working tool. The worst outcome isn't slow adoption — it's breaking the tool agents already rely on.

Section 9 Post-Launch Support
Weeks 1–4

Weekly office hours (30 min drop-in) for questions and feedback. PM monitors adoption dashboard daily — who's using it, who isn't, and why. Follow up personally with low-adoption agents.

Month 2+

Monthly check-ins. Feature request intake opens for Phase 2 prioritization. Begin measuring AHT and FCR impact with enough data for statistical significance.

Ongoing

Champions channel stays active. Quarterly satisfaction survey. Champions transition from support role to feature advocate — helping shape Phase 2 priorities based on daily usage patterns.

The best rollout is one where agents are asking "when can I get it?" before launch day. Every decision in this plan — from hand-picking alpha agents to keeping old tools accessible — is designed to earn trust, not mandate compliance. If the tool is clearly better, adoption follows.
