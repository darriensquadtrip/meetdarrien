Resourcing Philosophy

The conventional approach for internal tool MVPs is 6-8 weeks with a team of 5-6. That's too slow. Two weeks to get working software in front of real CSO agents. Three more weeks to polish, test, and roll out. Total: 5 weeks from kickoff to full deployment.

This isn't wishful thinking — it's a reflection of how modern product teams ship when they have clear requirements and use AI-assisted development. And it works because we're splitting work by component, not by layer. Each engineer owns their feature vertical end-to-end — database queries, API endpoints, and UI — so there are no handoffs between a "backend person" and a "frontend person." One engineer builds search and it works. The other builds panels and they work. They integrate at the data layer, not at an API contract boundary.

The trap is over-resourcing. More engineers doesn't mean faster at this scale — it means more coordination overhead. Justworks engineers are full-stack — they work across the entire stack within their product area, not in frontend/backend silos. Two full-stack engineers each owning a distinct feature vertical, plus a Data Services engineer for the Salesforce pipeline, can work in true parallel with minimal coordination. Keep the team small, the scope tight, and the feedback loops fast.

Software Engineer — Search & Identity
Full-stack · Dedicated · Full-time
LONG POLE WEEK 1 START
▼
Owns (Full Vertical)

Everything the agent sees first: ⌘K search, company profile card, caller identity, and the status indicator strip. Owns these features end-to-end — the database queries, the API endpoints, the caching layer, AND the React UI components. This is the "front door" of Customer Central: search → who is this person → what's their status. One engineer, one vertical, zero handoffs.

Week-by-Week
  • Days 1-2: Data access spikes — validate API connectivity to production DB (company profiles, member data, billing status) and Zendesk (for open ticket counts in the status strip). Confirm response shapes, auth, rate limits. Measure latency for each call — we need Tier 1 data in <500ms.
  • Days 3-5: Build the search endpoint and company profile API. Stand these up as runnable integration tests with measured response times — prove the data layer works before touching any UI.
  • Week 2: Build the full search UI, company profile card, caller card, and status strip. Wire to live APIs. End of week: an agent can search for a company and see a complete, real-data profile in under 500ms.
  • Weeks 3-5: Iterate based on alpha/beta feedback — search ranking, fuzzy matching, edge cases (new companies with no history, multiple contacts with same name). Performance tuning under concurrent load.
Why This Vertical Is Critical

Search is the entry point for every single interaction. If search is slow, nothing else matters. If the company profile is wrong, agent trust is destroyed on the first call. This engineer owns the highest-stakes, highest-frequency path in the product — and because they own it end-to-end (API through UI), they can optimize the full pipeline without waiting on anyone.

Software Engineer — Panels & History
Full-stack · Dedicated · Full-time
MOST FEATURES HEAVIEST DATA
▼
Owns (Full Vertical)

Everything the agent digs into after identifying the caller: interaction timeline, recent activity feed, and all problem-specific panels — pay stubs, tax docs, benefits, invoices, payment cascade. Owns these end-to-end — the Zendesk integration, the production DB pass-through queries, the API endpoints, AND the React UI (timeline component, tabbed panels, deep links). This is where the data complexity lives: cross-system merges, 4-tier loading, and the heaviest payloads.

Week-by-Week
  • Days 1-2: Data access spikes — validate Zendesk API for ticket retrieval and the production DB for payroll/billing data. Measure response times. The critical question: can we merge Zendesk tickets with Salesforce case data, and how fast?
  • Days 3-5: Build the interaction timeline API and the first problem panel endpoint (pay stubs). Stand these up as integration tests — run the API calls, log response times, flag anything over 500ms. Scaffold the app shell and component library alongside Eng 1.
  • Week 2: Build the timeline UI, activity feed, and tabbed problem panels. Wire to live APIs. Each panel follows the same pattern: API call → loading skeleton → render. End of week: an agent can see a full interaction history and click into any problem panel with real data.
  • Weeks 3-5: Iterate based on alpha/beta feedback — data accuracy in the timeline merge, loading performance for heavy panels (pay stubs at ~20KB), graceful degradation when Zendesk is slow. Performance tuning on the Tier 2/3/4 loading sequence.
Why This Vertical Is the Heaviest

This engineer owns the most features (6 of 9 MVP features) and touches the most data sources. The interaction timeline alone merges Zendesk and Salesforce records, the pay stub panel queries the production DB in real-time, and the status indicators hit all 5 sources in parallel. But because one person owns the full vertical — API through UI — they can optimize the loading sequence holistically: pre-fetch what's likely needed, lazy-load what's not, and cancel abandoned requests when the agent switches tabs. No one else needs to be in the loop for those decisions.

▼
Owns

Data accuracy testing — the single most critical QA concern. If the dashboard shows an incorrect billing status, wrong employee count, or stale interaction history, agent trust is destroyed immediately. Also owns integration testing across the Zendesk-Salesforce-production DB data flow.

Focus Areas
  • Data accuracy: Cross-reference dashboard data against source systems. Every data point shown to an agent must match reality.

Why Shared

QA ramps in Week 2 once there's something to test. They don't need to be full-time — they need focused bursts during integration testing and pre-launch validation. A shared resource with clear test plans is more effective than a dedicated QA engineer with idle time.

Why Component Ownership, Not Layer Ownership

The traditional approach splits work by layer: one person builds APIs, another builds UI, and they coordinate via API contracts. That creates a handoff bottleneck — the frontend waits for the backend, API contracts change mid-sprint, integration bugs fall between two owners. Instead, we split by component vertical: Engineer 1 owns Search & Identity end-to-end (DB → API → UI), Engineer 2 owns Panels & History end-to-end. Each engineer can validate their API calls, measure latency, and build UI without waiting on anyone. The Data Engineer from Data Services brings Salesforce pipeline expertise that feeds both verticals. This mirrors how Justworks already works — full-stack engineers owning product areas, not layers.

04
De-Risking Strategy

The biggest risk isn't building the wrong thing — it's discovering on Day 15 that a data source we assumed was accessible isn't. Every risk gets tested in Week 1.

Risk #1: Salesforce Integration Complexity

What could go wrong: Salesforce org has custom objects, non-standard relationships, or API rate limits that make real-time querying infeasible. The Zendesk-Salesforce shared key doesn't exist or is unreliable.

De-risk move: Day 1-2 paired spike with both engineers + Data Engineer + Caroline's team. Don't write production code — just prove we can pull Account, Contact, and Case data via API and join it to a Zendesk ticket. If we can't, we know on Day 2 and can adjust scope (e.g., Salesforce data in Phase 2, MVP uses only Zendesk + prod DB).

Risk #2: Data Freshness Expectations vs. Reality

What could go wrong: The production DB read replica has a 15-minute lag. Zendesk API rate limits constrain how frequently we can poll. Agents see stale data and lose trust.

De-risk move: Measure actual latency during Week 1 spikes. If data isn't fresh enough for live calls, show freshness timestamps on every data point ("Updated 3 min ago") so agents know what they're looking at. Transparent staleness is better than invisible staleness.

Risk #3: Agent Adoption

What could go wrong: Agents have built muscle memory around their current 6-tab workflow. Even with a better tool, changing behavior is hard. They open Customer Central, glance at it, and go back to Zendesk.

De-risk move: Alpha with 5 volunteer agents, not 5 assigned agents. Volunteers are intrinsically motivated to try the new thing. Daily feedback sessions in Week 3 catch adoption friction early. The gate question is explicit: "Are agents choosing to use this?"

Risk #4: Scope Creep During Alpha

What could go wrong: Alpha agents request 20 features. Team starts building them. Week 4 beta slips to Week 6. The 5-week plan becomes an 8-week plan.

De-risk move: Pre-commit: alpha feedback goes into three buckets. (1) Fix now: data accuracy issues, broken interactions, blocking problems. (2) Fix before beta: usability improvements that take <1 day. (3) Backlog: everything else. The PM's job during alpha is to say "great idea, that's Phase 2" more often than "let's build that."
